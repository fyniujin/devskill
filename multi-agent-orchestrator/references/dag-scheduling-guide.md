# DAG 调度深度指南

## 概述

本文档详细说明多Agent协作编排引擎的 DAG 调度机制，包括拓扑排序原理、执行策略、并行调度等内容。

---

## 拓扑排序

### 原理

拓扑排序（Topological Sort）是将有向无环图（DAG）的所有节点排成一个线性序列，使得对每条边 (u, v)，u 都排在 v 前面。

本引擎使用 **Kahn 算法**（基于入度）：

1. 计算每个节点的入度（有多少条边指向它）
2. 将入度为 0 的节点加入队列
3. 取出队首节点，将其所有下游节点的入度减 1
4. 若某下游节点入度变为 0，加入队列
5. 重复直到队列为空

### 示例

```
DAG:
  A → B → D
  A → C → D
  B → E

入度: A=0, B=1, C=1, D=2, E=1

排序过程:
  1. 取 A (入度0), B和C入度-1 → B=0, C=0
  2. 取 B (入度0), D和E入度-1 → D=1, E=0
  3. 取 C (入度0), D入度-1 → D=0
  4. 取 D (入度0)
  5. 取 E (入度0)

结果: A → B → C → D → E
```

### 代码实现位置

`scripts/dag_validator.py` 的 `topological_sort()` 函数。

## 执行策略

### 串行执行（默认）

按拓扑排序顺序逐个执行节点。每个节点必须等待所有依赖节点完成后才能执行。

**适用场景**：节点间有严格数据依赖、资源受限、需要严格顺序控制。

### 并行执行（同层并行）

拓扑排序的每一"层"（入度同时为 0 的节点）可以并行执行。

```
Layer 0: [A]           ← 串行执行
Layer 1: [B, C]        ← B 和 C 可并行
Layer 2: [D]           ← 等待 B 和 C 都完成
Layer 3: [E]           ← 等待 D 完成
```

**注意**：当前版本（1.0）仅支持串行执行。并行执行需要 AI 编排器自行管理并行子任务的 spawn 和回收。

## 依赖关系

### depends_on 字段

每个 Agent 节点的 `depends_on` 字段定义了其上游依赖：

```json
{
  "id": "analyzer",
  "depends_on": ["collector", "cleaner"]
}
```

含义：`analyzer` 必须在 `collector` 和 `cleaner` 都完成后才能执行。

### 状态传递

当节点完成时，其 `output_data` 会自动保存到 `pipeline_state.json`。下游节点执行时，引擎会自动注入所有上游节点的输出：

```
collector.output_data → analyzer 的输入参数
cleaner.output_data → analyzer 的输入参数
```

## 环检测

### 为什么需要

循环依赖会导致拓扑排序无法完成（部分节点永远不会入度为 0），执行时会陷入死锁。

### 检测方法

使用 Kahn 算法：如果排序后访问的节点数 < 总节点数，说明存在环。

### 修复方法

找到环路上的节点，移除其中一条依赖边，打破循环。

```
错误: A → B → C → A  (循环)
修复: A → B → C      (移除 C → A)
```

## 孤立节点

### 定义

没有上游依赖，也没有任何节点依赖它的节点。

### 影响

孤立节点不影响 DAG 的正确性，但通常意味着遗漏了依赖关系。

### 处理建议

- 检查是否遗漏了 `depends_on` 字段
- 确认该节点是否确实独立（如初始化任务）
- 如果是有意为之，可以在 DAG 中添加注释说明
