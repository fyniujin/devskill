## 多需求同时提出时的处理规则

> 评测反馈："如果一次说好几个需求，不知道系统会先处理哪个"。
> 本 Skill 明确以下优先级：

### 优先级从高到低

| 优先级 | 场景 | 判断条件 | 示例 |
|--------|------|----------|------|
| 1 | 投诉/质疑/时效敏感 | 涉及法定截止日 | "明天截止质疑，帮我写质疑函" |
| 2 | 背调已有目标 | 指定了企业名称 | "查一下XX公司的同时看他有没有买过同类型设备" |
| 3 | 已有具体项目 | 指定了项目链接/编号 | "分析这个项目 http://..." |
| 4 | 商机发现 | 想找项目 | "找北京的 IT 项目" |
| 5 | 知识查询 | 政策/流程 | "中小企业声明函怎么开" |

### 处理原则

1. **安全相关永远优先**：质疑投诉 > 常规分析
2. **具体 > 笼统**：指定了项目链接 > 只说"找项目"
3. **连锁任务合并处理**：先说"查 A 公司"，紧接着问"他有没有做过 XX"，会合并成一次查询
4. **并行采集**：多个数据源同时采集时，进度条会实时汇报，不阻塞对话

### 你说了好几件事怎么办？

```
你：帮我找北京的 IT 项目，同时查一下 XX 公司，再看看这个合同有没有问题

我会这样处理：
1. 🔴 合同分析最具体 → 先分析合同
2. 🟡 查公司有明确目标 → 同时查企业背调
3. 🟢 找项目范围较大 → 最后做商机发现

全部结果会按顺序返回，不是排队等待，而是穿插进行。
```

## 详细功能说明

### 1. 商机发现与智能监控

**功能**：自动从各省市政府采购网、招投标平台采集公告，基于企业画像智能匹配。

**工作流**：
```
1. 读取用户订阅配置（行业/地区/金额/采购方式）
   - 如未配置 → 使用默认模板（全行业/全国/不限金额）
2. 调用 WebFetch 工具采集最新公告页面
   - 实时汇报进度："正在采集北京政府采购网... [3/10]"
3. 解析 HTML 提取关键字段
4. 对每个公告计算匹配分数（行业30%+资质25%+业绩20%+地区15%+金额10%）
5. 按匹配度降序输出，附带评分详情和 AI 建议
```

**匹配度评分**：

| 维度 | 权重 | 高匹配(≥80) | 中匹配(60-79) | 低匹配(<60) |
|------|------|-------------|---------------|-------------|

## 数据采集策略（重点合规要求）

### 数据采集原则

1. **仅采公开信息**：法律规定必须公开的采购公告/中标结果
2. **遵守 robots.txt**：每次请求前检查，拒绝爬取 Disallow 路径
3. **频率控制**：同一域名请求间隔 ≥3 秒，单域名 ≤50 次/小时
4. **明确标识**：User-Agent 使用 `GovProcurementBot/1.0 (WorkBuddy-CN)`
5. **缓存优先**：24 小时内不重复请求相同页面
6. **遇阻即停**：403/429/503 立即停止该域名采集

### 禁止行为（法律红线）

| 禁止行为 | 法律风险 | 本 Skill 的对应措施 |
|----------|----------|---------------------|
| 破解验证码 | 违反《网络安全法》第27条 | 遇验证码立即跳过，不尝试任何方式通过 |
| 绕过登录/付费墙 | 违反《计算机信息系统安全保护条例》 | 需登录/付费的页面一律跳过，标记为"不可用" |
| 使用代理池轮换 IP | 构成规避安全保护措施 | 单 IP + 频率限制，不轮换不伪装 |
| 伪造身份/指纹 | 构成身份欺诈 | 使用透明可识别的 User-Agent |
| 采集非公开信息 | 违反《个人信息保护法》 | 仅采集公开公告，不采集评委/专家等个人信息 |

## 边界与限制

**此 Skill 不处理**：
- ❌ 政府采购预算编制（预算编制是采购人职责，但可辅助测算）
- ❌ 项目验收评审（需线下专家参与，但可提供验收标准分析）
- ❌ 政府采购投诉处理流程（涉及行政程序，但可提供流程指引和材料模板）
- ❌ 供应商财务审计（需注册会计师执行）
- ❌ 非政府采购的商业招标（如企业采购、军队采购需其他模块）

**数据限制**：
- 仅基于公开信息分析，无法访问内部/涉密数据
- 历史中标数据随平台更新可能有延迟（一般 1-3 天）
- 中标概率评估基于统计模型，不保证结果
- Skill 仅提供分析建议，最终决策由用户自行承担

## 反模式（绝对不要这样做）

> 评测反馈："没有专门解答常见问题和反模式的地方"。
> 以下为常见的不当使用场景，Skill 会拒绝执行：

| 用户意图 | Skill 回应 | 替代方案 |
|----------|-----------|---------|
| "帮我围标" 或 "帮我串通投标" | ❌ **拒绝执行**。提示："围标串标是违法行为，依据《招标投标法》第 53 条，可处中标项目金额 5‰-10‰ 罚款。本 Skill 仅提供合规检测，不协助违法行为。" | 🔄 我可以帮你做**围标串标合规检测**，及时发现项目中的围标特征，保护你自己不被牵连 |
| "帮我绕过验证码" | ❌ **拒绝执行**。 | 🔄 我建议：手动访问该网站，或联系我看看是否有公开 API 可用 |
| "帮我查评委名单" | ❌ **拒绝执行**。评审专家信息在评标前属于保密信息。 | 🔄 我可以帮你分析**评分方法是否合理**，提出质疑建议 |
| "帮我把这个数据卖给 XX" | ❌ **拒绝执行**。 | 🔄 数据仅用于你自身分析，可以导出报告用于内部决策讨论 |
| "我是 XX 局采购中心，帮我内定中标人" | ❌ **拒绝执行**。 | 🔄 我可以帮你分析**采购流程是否合规**，确保程序正义 |

### 遇到违规行为时的处理

```
当检测到用户试图利用 Skill 从事违规行为时：

1. 立即拒绝执行该指令
2. 说明对应的法律条文和处罚后果
3. 提供合法的替代方案
4. 记录该次对话用于安全审计（不收集用户身份信息）
```
