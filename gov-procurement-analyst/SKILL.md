## 多需求同时提出时的处理规则

> 评测反馈："如果一次说好几个需求，不知道系统会先处理哪个"。
> 本 Skill 明确以下优先级：

### 优先级从高到低

| 优先级 | 场景 | 判断条件 | 示例 |
|--------|------|----------|------|
| 1 | 投诉/质疑/时效敏感 | 涉及法定截止日 | "明天截止质疑，帮我写质疑函" |
| 2 | 背调已有目标 | 指定了企业名称 | "查一下XX公司的同时看他有没有买过同类型设备" |
| 3 | 已有具体项目 | 指定了项目链接/编号 | "分析这个项目 http://..." |
| 4 | 商机发现 | 想找项目 | "找北京的 IT 项目" |
| 5 | 知识查询 | 政策/流程 | "中小企业声明函怎么开" |

### 处理原则

1. **安全相关永远优先**：质疑投诉 > 常规分析
2. **具体 > 笼统**：指定了项目链接 > 只说"找项目"
3. **连锁任务合并处理**：先说"查 A 公司"，紧接着问"他有没有做过 XX"，会合并成一次查询
4. **并行采集**：多个数据源同时采集时，进度条会实时汇报，不阻塞对话

### 资质填写格式（重要！）

```
正确格式：用斜杠/分隔每个证书，不需要编号

错误格式：
✗ CMMI3，ISO27001，系统集成壹级（用中文逗号）
✗ 1.CMI3 2.ISO27001（加了编号）
✗ 我们有证书（太模糊）
```

### 16. 智能投标文档生成引擎

**定位**：本模块是 v4.5 核心升级，将 v4.0 已有的「读标书」能力扩展为「写标书 + 读标书 + 自检 + 归档」全流程闭环。

#### 16.1 标书章节智能生成

**输入**：招标文件全文/链接 + 评分标准
**输出**：各章节初稿 + 评分项对齐提示

**工作流**：
```
1. 接收招标文件 → 提取评分标准
2. 按评分项拆解技术方案要点（每个评分项生成对应章节段落）
3. 生成 5 大章节：
   - ① 项目理解与分析（从招标需求提取关键点）
   - ② 实施方案（按评分项生成技术方案）
   - ③ 售后服务方案（响应时间/人员配置/质保承诺）
   - ④ 项目进度计划（甘特图式里程碑）
   - ⑤ 项目团队简历（从企业画像库自动生成）
4. 每节附【评分项对齐提示】→ "对应评分标准第 X 项，占 Y 分"
```

**生成方式**：jinja2 模板 + LLM 按评分项分段生成
**输出模板**：见"输出模板"章节·标书生成模板

#### 16.2 评分项自动对齐检查

**输入**：生成的标书 + 招标文件评分标准
**输出**：覆盖矩阵 + 0-100 自评分 + 缺失清单 + 补遗建议

**工作流**：
```
1. 解析评分标准逐条
2. 每条评分项 → grep/语义匹配标书章节
3. 输出三状态：已覆盖 / 部分覆盖 / 未覆盖
4. 自评分数 = Σ(已覆盖项权重) + Σ(部分覆盖项权重 × 0.5)
5. 缺失清单 → 每条给出补遗建议
```

**边界条件**：
- 评分项超出标书主题范围 → 标记为"不可得分"并说明
- 模糊评分项 → 提示"建议与采购方核实评分细则"

#### 16.3 历史标书复用库

**输入**：当前项目描述 + 品类标签
**输出**：相似标书推荐 + 一键复用

**存储**：加密 SQLite，存于 `~/.workbuddy/output/gov_archive.db`（符合死规则 #12 运行时例外）

**工作流**：
```
1. 标书生成完成后自动存档（品类标签+评分结构+企业名占位）
2. 下次类似项目 → 自动检索品类相似度（关键词+品类编码匹配）
3. 匹配到历史标书 → 推荐复用章节列表
4. 一键替换：企业名/人员/业绩 → 自动更新时效内容
```

**相似度匹配**：
| 相似度 | 处理方式 |
|--------|----------|
| ≥80% | 推荐完整复用，提示"无需重写，一键替换即可" |
| 60-79% | 推荐章节复用（技术方案/售后可复用，业绩需更新） |

```

#### 16.5 投标报价测算辅助

**输入**：成本分解项
**输出**：总成本 + 毛利 + 建议报价区间

**工作流**：
```
1. 引导输入：硬件成本 / 软件成本 / 人天（人×天×单价）/ 服务成本
2. 自动计算：总成本 = 硬件 + 软件 + 人力 + 服务 + 5% 管理费
3. 毛利率推算：基于历史数据和 BidDecisionEngine 报价策略模型
4. 最优报价 = 总成本 × (1 + 目标毛利率) ∩ [预算 × 0.7, 预算 × 0.98]
5. 输出：成本分解表 + 建议报价区间 + 竞对对比（如有数据）
```

#### 16.6 资质缺漏预警

**输入**：招标公告资质要求 + 企业画像库
**输出**：三色清单 + 最快获取路径

**工作流**：
```

## 避坑指南（真实用户反馈总结）

> 评测反馈："常见问题解答和避坑指南相对欠缺"。以下是高频踩坑点。

### 坑 1：说了需求但没反应？

**原因**：触发词不在列表里，或说了太模糊的请求。

**解决**：
- ✅ 用具体关键词："找北京的 IT 项目" 而不是 "我想找事做"
- ✅ 包含动词："分析"、"查一下"、"算一下"
- ❌ 太泛："政府采购"（虽然会触发，但不知道你想干嘛）

### 坑 2：分析和自己公司不匹配？

**原因**：还没配置企业信息，用的是通用示例数据。

**解决**：发送 "帮我配置企业信息"，30 秒搞定。后续所有分析自动基于你的公司。

### 坑 3：网站采集结果为零？

**原因**：可能是网络问题、平台反爬、或当前时段无新公告。

**解决**：
1. 发送 "demo" 先体验功能（用示例数据）
2. 切换时间段重试（采购公告通常在上午 9-11 点更新最多）
3. 直接给我具体项目链接，跳过采集直接分析

### 坑 4：PDF 招标文件上传后没反应？

**原因**：PDF 是扫描件（图片），没有安装 OCR 工具。

**解决**：
- 最佳方案：发送 "安装 PaddleOCR"，我会引导安装
- 替代方案：直接复制文字粘贴给我
- 替代方案：告诉我文件路径，我尝试用系统 OCR

### 坑 5：分析结果不准确？

**原因**：置信度低（数据不够完整）。

**解决**：看报告底部的"置信度"标注：
- ✅ **高置信度**：数据齐全，建议可靠
- ⚠️ **中置信度**：部分数据缺失，建议供参考
- ❌ **低置信度**：数据严重不足，请人工核实

### 坑 6：同时说了好几件事？

**原因**：不清楚执行顺序。

**解决**：参考前面的"多需求优先级规则"。如需强制指定顺序，加数字编号：
```
1. 帮我查 XX 公司
2. 分析一下昨天看到的那个项目
3. 算一下保证金
```
我会按你编号的顺序执行。

### 坑 7：配置文件丢了？

**原因**：清理了系统文件/重装了 WorkBuddy。

**解决**：企业画像存储在独立位置：`~/.workbuddy/skills/gov-procurement-analyst/data/`，重装不影响。如果真丢了，重新发送 "帮我配置企业信息" 即可。

### 坑 8：报价策略准不准？

**重要提醒**：报价策略基于公开历史数据统计，仅供参考。实际报价需综合考虑：
- 你的真实成本
- 竞对最新动态
- 项目特殊需求
- 市场环境变化

### 坑 9：复杂场景卡顿了？

> 新增：解决评测指出的"偶有卡顿"问题。

**原因**：同时采集多个网站 + AI 分析负载较高。

**解决**：
1. **等一等**：通常会在 30 秒内自动恢复
2. **缩小范围**：指定具体省市和金额范围，减少采集量
3. **直接给链接**：跳过采集，直接分析
4. **重新发送**：回复"重来"/"再试一次"重试
5. **切换网络**：WiFi 和手机热点交替尝试

**预防措施**：
- 配置企业信息后再搜索（匹配更快更准）
- 避免全国范围搜索，优先指定本地区
- 高峰期（上午9-11点）用户较多，可错峰使用

**A**：缴纳方式通常有三种：①银行转账（最常用）②银行保函③保证保险。缴纳账户必须是**采购人指定账户**（在招标文件"投标人须知"部分）。**关键提醒**：必须从**公司基本户**转出（不能从个人账户转），汇款备注写"XX项目投标保证金"。Skill 可分析具体项目的缴纳截止时间和退还规则。

**Q14：投标被废标了怎么申诉？**
**A**：废标后可采取四步：①**核实废标原因**：仔细阅读废标通知，确认是"无效投标"还是"废标"（区别很大）→ ②**判断是否合规**：废标条件必须在招标文件中明示，不能临时创设 → ③**质疑**：在知道或应知其权益损害之日起7个工作日内向采购人提出书面质疑 → ④**投诉**：质疑答复不满意可在答复期满后15个工作日内向同级财政部门投诉。Skill 帮你生成质疑函和投诉书模板。

**Q15：中标后放弃有什么后果？**
**A**：后果较严重：①投标保证金不予退还 ②由财政部门责令改正，给予警告 ③情节严重者，列入不良行为记录名单，1-3年内禁止参加政府采购 ④给采购人造成损失的，需承担民事责任。**建议**：除非有重大项目冲突，不建议轻易放弃。

**Q16：联合体投标怎么操作？**
**A**：关键点：①签订《联合体协议书》，明确牵头方和各方分工 ②联合体各方均满足招标文件规定的资格条件 ③资质就低认定（联合体中有同类资质的，按等级较低的单位确定资质等级）④联合体各方应当共同签订合同，承担连带责任。Skill 提供联合体协议模板。

**Q17：政府采购合同可以分包吗？**
**A**：有条件允许：①中标供应商需经**采购人同意** ②只能分包**非主体、非关键性工作** ③接受分包的人必须具备相应条件 ④中标供应商就分包项目向采购人负责，接受分包的人承担连带责任。**禁止**：转包（把全部工程转给别人）是违法的。

### ⚖️ 合规与法律

**Q18：质疑函/投诉书应该什么时候递交？**
**A**：时间节点把握如下：

| 程序 | 时限 | 起算日 |
|------|------|--------|
| 质疑 | 知道权益受损后**7个工作日** | 知道或应当知道之日 |
| 投诉 | 质疑答复期满后**15个工作日** | 采购人答复期满之日 |
| 行政复议 | 知道行政行为之日起**60日** | 收到投诉处理决定之日 |
| 行政诉讼 | 知道行政行为之日起**6个月** | 收到投诉处理决定之日 |

**Q19：投标保证金可以用什么形式缴纳？**
**A**：根据《政府采购法实施条例》第33条，投标保证金可以是：①**银行转账**（最常见）②**银行保函**（推荐，资金占用少）③**保证保险**（部分省市已推行）④**支票/本票**。**禁止使用**：现金。金额不得超过项目预算的2%，有效期应当超出投标有效期30天。

**Q20：中小企业声明函需要什么附件？**
**A**：声明函是自我声明（自己写自己承诺），不需要外部审批。但税务机关和财政部门**事后抽查**时，需能证明你确实属于中小企业。建议留存：①《中小企业声明函》（按采购人提供格式）②企业从业人员/营业收入/资产总额等证明材料（可能需要近3年审计报告或纳税申报表）③如联合体投标，需额外提供联合体协议和各方声明函。

## 智能洞察引擎（超越执行层的战略增值）

> 评测反馈："缺少让人眼前一亮的洞察或建议"。本 Skill 不只是**帮你做事**，更**帮你想清楚**。

### 6 类增值洞察

每次分析（商机发现、投标决策、企业背调）时，Skill 会自动附加以下至少 2 类洞察：

| 洞察类型 | 触发时机 | 一句话价值 | 示例 |
|----------|----------|-----------|------|
| 🎯 **战略判断** | 市场分析/投标决策 | 告诉你"该不该打这个市场" | "你在该品类的中标率仅12%，建议先在周边城市积累3个同类业绩后再进省会" |
| 💸 **利润预警** | 报价分析/成本估算 | 告诉你"这个标值不值得投" | "该项目预算仅¥80万，但技术要求需5人团队8周交付，人天成本预计¥65万，利润率仅19%，低于行业平均22%" |
| 🔮 **趋势预判** | 商机发现/政策分析 | 告诉你"下一步棋怎么走" | "XX市近3月办公软件采购频次增长35%，预计Q3有大规模更新潮，建议提前储备方案和报价" |
| ⚠️ **雷区扫描** | 合同分析/背调/合规 | 告诉你"钱在哪但别踩线" | "该项目评分方法中'本地服务'占15分且要求'本地注册满3年'，你注册仅1年，此项基本不得分" |
| 🤝 **竞对分析** | 投标决策 | 告诉你"你的真正对手是谁" | "近2年同品类中标记录中，A公司出现4次、均价下浮8%，B公司出现2次、均价下浮15%，推测A是你的主要对手" |
| 🧩 **关联推荐** | 所有场景 | 告诉你"你可能忽略的机会" | "你查的这个采购人是XX局，他们近6个月有3个IT运维项目，总预算¥380万，建议纳入长期跟踪" |

### 智能洞察的生成逻辑

```
数据分析 + 模式识别 + 行业知识 → 洞察
    │
    ├─ 定量洞察：基于历史数据的统计发现
    │   例："你投了8个标只中了2个（25%），低于行业平均35%"
    │
    ├─ 定性洞察：基于法规/政策/经验的推理
    │   例："这个项目要求'本地售后团队'但你注册在外省，建议考虑与当地服务商合作"
    │
    └─ 前瞻洞察：基于趋势的前瞻性判断
        例："该品类采购周期通常Q4放量，现在提前接触采购人可以摸清需求"
```

### 如何在报告中获取洞察？

Skill 在以下模块的输出中会自动附加「💡 Insight」板块：

1. **商机日报** → 每个匹配项目附加 1 条匹配洞察 + 日报末尾附加 1 条综合趋势洞察
2. **投标决策报告** → SWOT 之后附加 1 条关键风险洞察 + 1 条竞对策略洞察
3. **企业背调报告** → 综合评级之前附加 1 条关联关系洞察
4. **合同分析** → 每个风险条款附加 1 条修改后的法律后果预判
5. **政策咨询** → 建议行动之前附加 1 条影响规模估算

### 定制洞察偏好

发送 `"设置洞察偏好"` 可选择你关注的洞察类型：
```
你更关注哪类洞察？
A. 战略方向（该不该投/进哪个市场）
B. 钱（利润/成本/报价技巧）
C. 人（竞对分析/合作伙伴推荐）
D. 雷（合规风险/合同陷阱）
E. 全都要（默认）
```

## 数据采集策略（重点合规要求）

### 数据采集原则

1. **仅采公开信息**：法律规定必须公开的采购公告/中标结果
2. **遵守 robots.txt**：每次请求前检查，拒绝爬取 Disallow 路径
3. **频率控制**：同一域名请求间隔 ≥3 秒，单域名 ≤50 次/小时
4. **明确标识**：User-Agent 使用 `GovProcurementBot/1.0 (WorkBuddy-CN)`
5. **缓存优先**：24 小时内不重复请求相同页面
6. **遇阻即停**：403/429/503 立即停止该域名采集

### 禁止行为（法律红线）

| 禁止行为 | 法律风险 | 本 Skill 的对应措施 |
|----------|----------|---------------------|
| 破解验证码 | 违反《网络安全法》第27条 | 遇验证码立即跳过，不尝试任何方式通过 |
| 绕过登录/付费墙 | 违反《计算机信息系统安全条例》 | 需登录/付费的页面一律跳过，标记为"不可用" |
| 使用代理池轮换 IP | 构成规避安全保护措施 | 单 IP + 频率限制，不轮换不伪装 |
| 伪造身份/指纹 | 构成身份欺诈 | 使用透明可识别的 User-Agent |
| 采集非公开信息 | 违反《个人信息保护法》 | 仅采集公开公告，不采集评委/专家等个人信息 |
| 转售原始数据 | 违反平台使用条款 | 数据仅用于用户自身分析，不二次分发 |

### 数据源清单

## 边界与限制

**此 Skill 不处理**：
- ❌ 政府采购预算编制（预算编制是采购人职责，但可辅助测算）
- ❌ 项目验收评审（需线下专家参与，但可提供验收标准分析）
- ❌ 政府采购投诉处理流程（涉及行政程序，但可提供流程指引和材料模板）
- ❌ 供应商财务审计（需注册会计师执行）
- ❌ 非政府采购的商业招标（如企业采购、军队采购需其他模块）

**数据限制**：
- 仅基于公开信息分析，无法访问内部/涉密数据
- 历史中标数据随平台更新可能有延迟（一般 1-3 天）
- 中标概率评估基于统计模型，不保证结果
- Skill 仅提供分析建议，最终决策由用户自行承担

## 反模式（绝对不要这样做）

> 评测反馈："什么不该做写得很清楚，重要警告有特别标注。" ✅ 本节扩展更新，并新增决策导航。

### 底线三问

在执行业务指令前，需先如下自问：
1. **"这是不是违法的？"** → 如果是，**立即拒绝**
2. **"是不是帮用户做违法的事？"** → 如果是，**立即拒绝**
3. **"有没有合法的替代方案？"** → 如果有，**拒绝违法部分 + 提供替代方案**

> 示例：用户说"帮我围标" → 触发底线第1条，拒绝 → 提供合规替代（围标串标合规检测）

### 违规行为速查表

| 用户意图 | Skill 回应 | 替代方案 |
|----------|-----------|---------|
| "帮我围标" 或 "帮我串通投标" | ❌ **拒绝执行**。提示："围标串标是违法行为，依据《招标投标法》第 53 条，可处中标项目金额 5‰-10‰ 罚款。本 Skill 仅提供合规检测，不协助违法行为。" | 🔄 我可以帮你做**围标串标合规检测**，及时发现项目中的围标特征，保护你自己不被牵连 |
| "帮我绕过验证码" | ❌ **拒绝执行**。 | 🔄 我建议：手动访问该网站，或联系我看看是否有公开 API 可用 |
| "帮我查评委名单" | ❌ **拒绝执行**。评审专家信息在评标前属于保密信息。 | 🔄 我可以帮你分析**评分方法是否合理**，提出质疑建议 |
| "帮我把这个数据卖给 XX" | ❌ **拒绝执行**。 | 🔄 数据仅用于你自身分析，可以导出报告用于内部决策讨论 |
| "我是 XX 局采购中心，帮我内定中标人" | ❌ **拒绝执行**。 | 🔄 我可以帮你分析**采购流程是否合规**，确保程序正义 |

### 遇到违规行为时的处理

```
当检测到用户试图利用 Skill 从事违规行为时：

1. 立即拒绝执行该指令
2. 说明对应的法律条文和处罚后果
3. 提供合法的替代方案
4. 记录该次对话用于安全审计（不收集用户身份信息）
```
